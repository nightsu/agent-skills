# Figma Emit Spec

## Position in figma-workflow

本 skill 是 `figma-workflow-suite` 的一部分,处于 **phase E**(最后一环):

```
phase A → phase B → phase C-up → phase C-low → phase D → [phase E: figma-emit-spec] → handoff 出口
                                                                                      ├── builtin(task-breakdown.md)
                                                                                      ├── superpowers:writing-plans
                                                                                      ├── manual(用户自己接手)
                                                                                      └── pause(用户改完 open-questions 再回来)
```

可以通过两种方式调用:
- **由 orchestrator 调用** —— `figma-workflow feature=<name>` 在 phase E 自动路由到本 skill
- **独立调用** —— 用户直接调用本 skill,传入 `feature` 参数

## Prerequisites

执行本 skill 前,以下 5 份上游产物必须**全部存在且非占位**:

- `docs/design/<feature>/clarified-requirement.md`(phase A)
- `docs/design/<feature>/ui-understanding.md`(phase B)
- `docs/design/<feature>/api-mapping.md`(phase C-up)
- `docs/design/<feature>/component-mapping.md`(phase C-low)
- `docs/design/<feature>/design-token-patch.md`(phase D)

如有任一缺失,**拒绝执行**,提示用户先完成对应 phase。

## Calling convention

```
figma-emit-spec feature=<feature-name>
```

`feature=` 必传。本 skill 不需要任何 Figma 输入(纯合成,不调 Figma MCP)。

## 适用场景

- 已完成 phase A~D 五份产物,需要合并为后续 planning/spec authoring 主输入
- 跨产物存在 label / 字段 / module / token 冲突,需要自动校正或暴露给用户
- 准备进入 OpenSpec / planning / task breakdown 等实施准备阶段,需要决定 handoff 路径(builtin task / superpowers / manual / pause)

## 目标

本 skill **合成而非生产**:
- 重组 5 份上游产物的事实
- 自动校正 `label_drift` 冲突(以 phase D 真实 label 为准)
- 暴露 `field_unbound` / `module_missing_token` / `module_drift` 冲突到 open-questions
- 识别 `[deferred]` 标记的 open question,在 spec 中显式标"本期不实现"
- 在出口处让用户选择 handoff 路径
- 产出 `implementation-spec.md`,作为后续 planning/spec authoring 的唯一主输入
- 保持 Phase E review gate 和 handoff 边界,本 skill 不触发代码修改

**不**承担:
- 调用 Figma MCP(纯合成)
- 修改上游产物(只读)
- 自动回答 open questions
- 生成实施代码(那是 coding agent 的事)
- 调用 apply/coding agent
- 修改业务代码
- 生成 OpenSpec proposal

## 工作流

1. **读取 5 份上游产物**
   - 全部存在且非占位才继续,任一缺失→拒绝执行

2. **检测 4 种 cross-product conflict**
   - `field_unbound`:`api-mapping.md` 字段未在 `component-mapping.md` 被绑定 → 写入 open-questions
   - `module_missing_token`:`component-mapping.md` 的 module 未在 `design-token-patch.md` 抽 token → 写入 open-questions
   - `module_drift`:`ui-understanding.md` 的 module 未在 `component-mapping.md` 出现 → 写入 open-questions
   - `label_drift`:`component-mapping.md` 槽位 label ≠ `design-token-patch.md` 真实 label → **自动以 D 为准**,在 implementation-spec metadata 段说明,**不写入 open-questions**(答案已知)
   - 详细算法见 [references/conflict-detection.md](references/conflict-detection.md)

3. **识别 `[deferred]` open questions**
   - 扫描所有上游 .md 的 open questions / Open Questions 段
   - 含 `[deferred]` 前缀的项 → 在 implementation-spec 对应 Module 段显式标"本期不实现"
   - 不含 `[deferred]` 的 → 汇总到 open-questions.md 顶部

4. **合成 implementation-spec.md**
   - 按 [references/spec-template.md](references/spec-template.md) 的填充规则,逐章节从对应上游产物抽取内容
   - 任何新增/推断内容必须显式标 `(INFERRED)`
   - 不要引用 raw Figma JSON

5. **合成 open-questions.md**
   - 按来源分段:`From Phase A` / `From Phase B` / `From Phase C-up` / `From Phase C-low` / `From Phase D` / `Cross-Product Conflicts (auto-detected)`
   - 自动校正的 `label_drift` **不**出现在此处(已自动处理)

6. **写产物**
   - `docs/design/<feature>/implementation-spec.md`
   - `docs/design/<feature>/open-questions.md`
   - `docs/design/<feature>/inputs.md` 追加一条(含 emit_version + label_drift 校正数 + open-questions 总数)

7. **handoff 出口**(见 spec §5.2)
   - 检测环境(superpowers 是否安装)
   - 弹菜单 4 个选项让用户选
   - 选 builtin → 在 emit-spec 内部跑 task-breakdown 子流程,只产出 `task-breakdown.md`
   - 选 superpowers → 推荐在可用时选择;调用 `Skill("superpowers:writing-plans")`,只把 `implementation-spec.md` 作为输入
   - 选 manual → 退出并指向 `implementation-spec.md`;可提示外部目标包括 OpenSpec,但本 skill 不生成 OpenSpec proposal
   - 选 pause → 退出,用户回答 open questions 或编辑上游产物后重新调用本 skill
   - 用户选择保存到 `.workflow-prefs.json`(builtin / superpowers 时保存;manual / pause 不保存)

handoff 发生在 Phase E review gate 选择 Proceed 之后。handoff 后可以进入 OpenSpec / planning / task breakdown 等准备阶段;业务代码只能在用户明确确认执行 coding 后开始。

## Implementation Constraints
- 不直接复刻 Figma 图层层级
- 不将装饰性节点实现为独立组件
- 重复列表项使用数据驱动渲染
- 优先复用项目已有组件 / tailwind / css_module
- 样式 token 来自 design-token-patch.md
- API 字段以 api-mapping.md 为准

- [ ] 复用项目已有组件 / 框架

## 关键行为

1. **合成时不引入新事实**,只重组上游产物的事实
2. 任何新增/推断内容必须显式标 `(INFERRED)`
3. `label_drift` 自动校正(以 D 为准),不写入 open-questions
4. `[deferred]` 前缀的 open question 在 spec 对应 Module 段标"本期不实现"
5. spec 中**绝对不引用** raw Figma JSON
6. 不修改上游产物

## 自查规则

skill 在产物落盘后,输出自查报告供 orchestrator review gate 使用:

| 项 | 阈值 | 提示文案 |
|---|---|---|
| Open Questions 总数(不含 deferred) | > 0 | "N 个 open question 未解决 —— 强烈建议先回答再 handoff" |
| `label_drift` 自动校正数 | > 0 | ℹ "N 个 label 已自动校正(C-low → D)" |
| `field_unbound` + `module_missing_token` + `module_drift` 总数 | > 0 | ❌ "N 个跨产物冲突 —— 见 open-questions.md ## Cross-Product Conflicts" |
| Deferred items 数 | > 0 | ℹ "N 个 deferred items,本期不实现" |
| Verification Checklist 未勾选项 | > 0 | "Verification checklist 有 N 项待勾选" |
| INFERRED 标记数 | > 0 | "spec 中有 N 处 INFERRED 内容,建议人工核对" |

自查**不阻塞**产物落盘。

## 错误处理

| 情况 | 处理 |
|---|---|
| 任一上游产物缺失 | 拒绝执行,告知用户缺哪份 |
| 上游产物 markdown 解析失败(格式严重崩坏) | 透传错误,产物不落盘 |
| 跨产物冲突 | 写入 open-questions,review gate 标红;不阻塞产物落盘(label_drift 自动校正不算阻塞) |
| handoff 时 superpowers 检测失败但用户选了选项 2 | 透传"未检测到"错误,退回菜单 |

## 不要做的事

- ❌ 不调用 Figma MCP
- ❌ 不联网
- ❌ 在 spec 里**绝对不引用** raw Figma JSON
- ❌ 不修改上游产物
- ❌ 不修改业务代码
- ❌ 不调用 apply/coding agent
- ❌ 不生成 OpenSpec proposal
- ❌ 不自动答 open questions
- ❌ 不在 handoff = manual / pause 时产出 `task-breakdown.md`
- ❌ 不把 `label_drift` 写入 open-questions(已自动校正)
- ❌ 不把 `[deferred]` 视为 blocker

## 下游衔接

本 skill 的产物 `implementation-spec.md` 是后续 planning/spec authoring 的**唯一主输入**。业务代码不由 emit-spec 自己编写,只能在用户明确确认执行 coding 后开始。

- **handoff = builtin** → 轻量 fallback,同时产出 `task-breakdown.md`,用户拿去自己规划或实现
- **handoff = superpowers** → 可用时优先推荐,`superpowers:writing-plans` 把 spec 转 implementation plan,后续是否 coding 由用户确认
- **handoff = manual** → 用户自己拿 spec 去 OpenSpec / Cursor / Codex 等外部目标
- **handoff = pause** → 用户改完 open-questions 重新调用本 skill 回到 review gate

所有 handoff 路径都发生在 Phase E review gate Proceed 之后,且不会由 `figma-emit-spec` 写业务代码。

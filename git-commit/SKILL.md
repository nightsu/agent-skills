# Git Commit

## 目标

把当前工作区的变更整理成一个安全、最小、可解释的 git commit。

## 适用时机

- 用户说“提交一下”“帮我 commit”“整理成一个提交”
- 用户要先看当前改动是否适合提交
- 用户要把互不相关的改动拆成多个 commit
- 用户要先生成 commit message 再提交

## 工作流

1. 识别范围
   - 看 `git status --short`
   - 看 `git diff --stat`
   - 必要时看最近提交风格：`git log -5 --pretty=format:%s`
2. 划分提交
   - 默认只提交与当前任务相关的文件
   - 保留无关修改、未跟踪文件和用户现有工作
   - 如果一个工作区同时包含多个互不相关的逻辑改动，优先拆成多个 commit；如果无法判断，先问用户
3. 检查风险
   - 发现 secrets、环境文件、密钥、二进制、大体积生成物时，先停下来确认
   - 不要为了“干净”而删除或覆盖用户未要求的内容
4. 组织暂存
   - 优先使用明确的文件列表 `git add <files>`
   - 只有在范围明确且用户希望整体提交时，才考虑更大范围的 add
   - 混杂改动多时，用 `git add -p` 或拆分提交
5. 提交前验证
   - 检查 `git diff --cached --check`
   - 复核 staged diff 是否只包含本次任务
6. 写 commit message
   - 简洁、具体、命令式
   - 默认跟随仓库已有风格；若无风格，使用简短 conventional commit
   - subject 不要空泛，尽量包含 scope
7. 完成提交
   - `git commit -m "..."`
   - 如果 hook 失败，只修复与本次提交直接相关的问题
8. 反馈
   - 返回 commit hash、message、提交文件
   - 说明是否还有未提交的无关改动

## 规则

- 不要重置、回滚或覆盖用户改动，除非用户明确要求
- 不要默认 `git add .`
- 不要把无关文件混进一个提交
- 如果提交前需要运行测试，优先只跑成本低且和改动直接相关的检查
- 如果当前不是 git repo，明确说明

## 输出风格

- 先给结论，再给简短说明
- 只展示和提交相关的信息

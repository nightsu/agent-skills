# 项目面试分析器

## 目标

分析项目目录，并把它整理成适合面试使用的中文材料。

每个项目都要产出：

- 一份技术与业务介绍文档
- 一份面试讲述文档

默认还要补充两条可讲的演进方向：

- 理想设计：如果不受当前实现约束，应该如何重新建模和实现
- 渐进优化：如果必须基于现有代码演进，优先怎么拆、怎么稳、怎么重构

如果一次提供多个项目，再额外产出一份项目总索引，用于比较和排序。

## 工作流

1. 识别项目边界。
   - 把每个顶层仓库或子项目当成一个独立分析单元。
   - 如果一个目录里包含多个 app 或 package，要拆开并明确标注。
2. 做通用广扫。
   - 阅读 `README`、`package.json`、配置文件、入口文件、services、state 模块、路由、页面、测试、部署文件。
   - 优先基于源码和配置，而不是凭经验猜测。
   - 先阅读 [通用检查清单](references/common-checklist.md)。
3. 判断项目形态。
   - 先判断更像哪一类：
     - 业务应用：有页面、路由、服务层、用户流程
     - 组件库 / 设计系统：以组件、tokens、文档站、stories、测试矩阵为核心
     - 基建 / 平台 / 工具：以构建、发布、脚手架、SDK、CLI、平台能力为核心
   - 再按需加载对应 reference：
     - 业务应用：读 [application.md](references/application.md)
     - 组件库：读 [component-library.md](references/component-library.md)
     - 基建：读 [infrastructure.md](references/infrastructure.md)
4. 如果是业务应用，再判断更偏 B 端还是 C 端。
   - B 端：优先读 [b-side.md](references/b-side.md)
   - C 端：优先读 [c-side.md](references/c-side.md)
   - 混合型项目可以同时参考多份，但只加载真正相关的部分。
5. 做第二轮补扫。
   - 不要只停留在系统级基础设施。
   - 至少补读 2-4 个复杂实现，优先选择：
     - 多步骤编辑器 / 向导页
     - 嵌套表单 / 可编辑表格
     - 树 / 大纲 / 编排页
     - 规则试算 / 审批 / 结算 / 高约束页面
     - 明显依赖复杂交互的页面
6. 提炼两层信息。
   - 技术/业务层：项目做什么、谁在用、怎么运作、实现上有什么特别之处。
   - 面试层：怎么讲故事、先讲什么、有哪些 tradeoff、面试官可能怎么追问。
7. 明确不确定性。
   - 所有推断都标记为“推测”或 `inferred`。
   - 任何没有证据支撑的职责或结果都要明确写出需要用户确认。
8. 输出中文 Markdown。
   - 文档要简洁但有信息量。
   - 优先结构化 bullet，而不是长篇散文。

## 覆盖护栏

为了避免只抓到主线而漏掉实现最复杂的部分，在完成第一轮扫描后，必须额外执行下面的覆盖检查：

1. 先列出候选复杂模块。
   - 至少从文件名或目录中筛出这些候选：
     - `Edit*`、`Create*`、`Configure*`、`*Modal`、`*Management`
     - `*Config`、`*Outline`、`*Builder`、`*Editor`、`*Wizard`
     - 任何明显带有 steps / table-edit / drag-sort / nested-form / preview / batch-operation 特征的页面
2. 对候选模块做二次筛选。
   - 优先深读同时满足以下任一条件的模块：
     - 嵌套表单或可编辑表格很多
     - 创建 / 编辑 / 复制多态共存
     - 前后端结构需要转换
     - 一个页面下挂多个子组件目录
     - 交互状态明显复杂，例如拖拽排序、分步切换、联动预览、展开行、悬浮详情、批量操作确认流、资源锁
     - 涉及规则计算、金额计算、库存/状态联动、树结构编辑
3. 每个补扫模块，至少要读：
   - 页面入口文件
   - 对应 service 文件
   - 至少 1-2 个关键子组件或工具文件
4. 在最终输出前，显式自检：
   - 当前仓库里是否还有明显复杂但未深读的高交互编辑页、规则页、编排页或多态表单页？
   - 如果答案是“有”，优先补读再写结论。

如果项目规模较大，允许第一轮不深挖所有业务模块，但最终交付前必须保证：

- 至少 1 个系统级抽象案例
- 至少 1 个复杂业务建模 / 编辑器案例
- 至少 1 个带明显 tradeoff 的案例

不要把这三类都用同一类基础设施案例替代。

## 深挖要求

对每个项目，不能只停留在“功能清单”和“模块罗列”，还要把最重要的 1-3 个点讲深。每个重点点都尽量回答下面这些问题：

- 这个点到底解决了什么真实问题？
- 为什么它比“页面内直接写逻辑”或“常规实现”更好？
- 它的设计目标是什么？
- 它是怎么实现的？关键模块 / Hook / 服务层分别做了什么？
- 这里的 tradeoff 是什么？牺牲了什么，换来了什么？
- 面试官会怎么追问？应该怎么答？

如果仓库里存在以下实现特征，默认它们值得进入候选清单，不应只因为第一轮主线已经足够成立就跳过：

- 多步骤编辑器 / Step 表单
- 大纲编辑器 / 树编辑器 / 分层结构编辑器
- 异构内容编排页（一个容器下承载多种 itemType / schema / attachInfo）
- 规则试算、审批、金额计算、售后、结算等高业务约束页面
- 多模式表单（例如 create / modify / copy，或多种 type / mode / strategy）
- 大量子组件协同的复杂 Modal
- 明显依赖复杂交互的页面，例如：
  - 拖拽排序 / DragSort / SortButton
  - EditableProTable / 可编辑嵌套表格
  - 展开行 / 折叠面板 / 分层编辑
  - hover 详情 / 预览浮层 / 侧边抽屉
  - 批量选择、批量导出、批量开关状态
  - 异步确认流（先预览、再确认、再轮询结果）
  - 并发编辑保护、资源锁、离开拦截、beforeunload

## 复杂交互补扫要求

除了扫描“复杂业务建模”，还要单独检查项目里是否存在值得面试讲的“复杂交互设计”。不要把交互只理解为样式或动画，重点关注那些会改变状态结构、用户路径或错误率的交互机制。

优先补扫以下交互：

1. 拖拽 / 排序 / 重排
2. 可编辑表格 / 嵌套编辑器
3. 分步式流程
4. 预览驱动交互
5. 展开 / 折叠 / hover 详情
6. 批量操作与异步反馈
7. 并发保护与误操作保护

当一个交互点满足以下任一条件时，应该把它提升为候选亮点，而不是只当“UI 细节”略过：

- 交互本身改变了数据结构或状态管理方式
- 交互需要多个组件协作完成
- 交互背后有明显的错误预防、并发保护或一致性设计
- 交互同时绑定异步请求、局部状态和用户反馈

分析复杂交互时，尽量补充：

- 用户为什么需要这个交互，而不是普通表单提交
- 交互前后的状态如何变化
- 哪些操作容易误触或出错，代码里如何防止
- 这个交互牺牲了什么直观性，换来了什么效率或正确性
- 面试时如何把它从“页面效果”讲成“交互设计与状态治理”

## 双轨输出要求

对每个足够复杂、值得讲设计的项目，优先同时给出两条路线：

1. 理想设计
   - 适合回答“如果重做一次，你会怎么设计？”
   - 重点讲领域建模、状态边界、协议设计、解耦方式、可扩展性。
2. 渐进优化
   - 适合回答“在当前实现基础上，怎么优化更好？”
   - 重点讲低风险拆分、容错补齐、职责收敛、重复逻辑消除、订阅/副作用治理。

如果项目本身很简单，或者用户只是想要概要介绍，可以不强行展开两条路线，但一旦项目存在明显的复杂业务链路、状态同步、事件桥接、认证跳转、动态表单、SSE 或多入口复用，默认都应该产出这两条路线中的至少一条，并明确说明为何只给其中一条。

如果一个点满足以下任一条件，必须深挖并画图：

- 影响多个页面、多个业务入口、多个客户端或多个系统边界
- 同时包含状态判断、异步请求和副作用
- 涉及认证、授权、跳转、嵌入容器、分页、动态表单、复杂参数拼装、事件回调或状态同步
- 设计上明显存在“统一封装 vs 分散实现”的取舍

如果一个点只是“有个组件/有个 Hook/有个服务”，但没有跨入口复用、跨系统协同或设计判断的价值，不要把它包装成亮点。

## 输出要求

每个项目都输出：

- `01-technical-business-overview.md`
- `02-interview-story.md`

如果是多个项目，再补充：

- `00-project-index.md`

技术与业务介绍文档、面试讲述文档的结构参见 [输出模板](references/output-templates.md)。

## 产出检查

在交付前，至少检查以下事项：

- 是否已经区分了事实和推测。
- 是否至少有 1 个点写到了“设计目标 + 实现机制 + tradeoff”。
- 是否已经识别出真正的抽象层，而不是把普通组件误判成亮点。
- 是否有足够的证据支撑最强亮点。
- 如果存在复杂链路，是否加了 Mermaid 或等价的流程表达。
- 是否补充了理想设计与渐进优化两条演进方向，或明确说明不展开的原因。
- 是否已经同时覆盖：
  - 通用主线
  - 适合该项目形态的特定关注点
  - 至少一个复杂实现或复杂交互案例

## 质量要求

- 不要编造没有证据支撑的职责或结果。
- 尽量把结论绑定到具体文件、模块或配置。
- 多用具体模块、配置和流程，少用空泛赞美。
- 目标是面试可用性，不是 code review 完整性。

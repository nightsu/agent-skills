# 项目面试分析器

## 目标

分析项目目录，并把它整理成适合面试使用的中文材料。

每个项目都要产出：

- 一份技术与业务介绍文档
- 一份面试讲述文档

如果一次提供多个项目，再额外产出一份项目总索引，用于比较和排序。

## 工作流

1. 识别项目边界。
   - 把每个顶层仓库或子项目当成一个独立分析单元。
   - 如果一个目录里包含多个 app 或 package，要拆开并明确标注。
2. 扫描证据。
   - 阅读 `README`、`package.json`、配置文件、入口文件、services、state 模块、路由、页面、测试、部署文件。
   - 优先基于源码和配置，而不是凭经验猜测。
3. 提炼两层信息。
   - 技术/业务层：项目做什么、谁在用、怎么运作、实现上有什么特别之处。
   - 面试层：怎么讲故事、先讲什么、有哪些 tradeoff、面试官可能怎么追问。
4. 明确不确定性。
   - 所有推断都标记为“推测”或 `inferred`。
   - 任何没有证据支撑的职责或结果都要明确写出需要用户确认。
5. 输出中文 Markdown。
   - 文档要简洁但有信息量。
   - 优先结构化 bullet，而不是长篇散文。

## 深挖要求

对每个项目，不能只停留在“功能清单”和“模块罗列”，还要把最重要的 1-3 个点讲深。每个重点点都尽量回答下面这些问题：

- 这个点到底解决了什么真实问题？
- 为什么它比“页面内直接写逻辑”或“常规实现”更好？
- 它的设计目标是什么？
- 它是怎么实现的？关键模块 / Hook / 服务层分别做了什么？
- 这里的 tradeoff 是什么？牺牲了什么，换来了什么？
- 面试官会怎么追问？应该怎么答？

如果项目里存在明显的复杂业务链路、认证/授权、跳转、嵌入容器、动态表单、分页、权限、多入口复用、异步回调、事件桥接或状态同步，优先把这些点作为“亮点 / 难点”深挖对象。

当你识别到一个亮点时，不要只写“封装了 X”，要至少补充：

- 设计思路
- 实现机制
- 风险点 / 边界条件
- 复用方式
- 面试表达方式

如果链路比较长，建议在输出中补一张 Mermaid 图，帮助说明状态流、跳转流或数据流。

## 深挖方法

把每个重点点按下面顺序拆开，避免只写结论：

1. 问题定义
   - 这个模块/流程到底卡在什么真实业务问题上？
   - 如果不做这个抽象，页面内会出现哪些重复、分散或高风险逻辑？
2. 设计目标
   - 这个设计优先解决的是复用、稳定性、性能、还是开发体验？
   - 这里有没有明确的约束，例如 tabBar、WebView、登录态、回退栈、运营配置？
3. 关键机制
   - 输入是什么，输出是什么，中间怎么处理？
   - 哪个模块负责校验，哪个模块负责调度，哪个模块负责副作用？
4. 边界与失败场景
   - 什么情况下会失效？
   - 怎么兜底、怎么重试、怎么避免重复执行？
5. tradeoff
   - 统一抽象牺牲了什么灵活性？
   - 配置驱动牺牲了什么直观性？
6. 复用范围
   - 哪些页面/模块复用了它？
   - 这个抽象是不是被多个入口依赖？
7. 面试表达
   - 用一句话概括这个设计。
   - 再用“问题 -> 设计 -> 结果”的顺序讲一遍。

如果一个点满足以下任一条件，必须深挖并画图：

- 影响多个页面、多个业务入口、多个客户端或多个系统边界
- 同时包含状态判断、异步请求和副作用
- 涉及认证、授权、跳转、嵌入容器、分页、动态表单、复杂参数拼装、事件回调或状态同步
- 设计上明显存在“统一封装 vs 分散实现”的取舍

如果一个点只是“有个组件/有个 Hook/有个服务”，但没有跨入口复用、跨系统协同或设计判断的价值，不要把它包装成亮点。

## 输出要求

每个项目都输出：

- `01-technical-business-overview.md`
- `02-interview-story.md`

如果是多个项目，再补充：

- `00-project-index.md`

## 技术与业务介绍文档

内容包括：

- 一句话概述
- 业务背景
- 使用者 / 相关方
- 技术栈
- 架构 / 仓库结构
- 设计思路
- 关键流程
- 亮点 / 难点的设计思路
- 难点 / 风险点
- 证据
- 未确认信息 / 假设

补充要求：

- 对“亮点 / 难点”不要只写结论，要写出设计原因和实现方式。
- 对复杂链路，优先使用项目内证据串起流程，再用 Mermaid 或分步 bullet 表达。
- 如果有明显的复用抽象，说明抽象边界是什么、被哪些页面复用、为什么这么抽。
- 如果一个抽象层的实现比较关键，最好写出它的输入、输出、缓存/状态、失败处理和复用点。

## 面试讲述文档

内容包括：

- 1 分钟介绍
- 我可能负责的内容
- 最强亮点
- 亮点 / 难点的设计思路
- 最难的问题
- tradeoff
- 影响 / 价值
- 可能的追问
- 简短回答角度
- 简历 bullet 版本

补充要求：

- “最强亮点”必须能落到具体模块、文件或流程。
- “最难的问题”要尽量围绕设计决策、边界条件和失败场景。
- 如果一个亮点本质上是一个抽象层，要明确写出它的输入、输出、内部处理和复用方式。

## 产出检查

在交付前，至少检查以下事项：

- 是否已经区分了事实和推测。
- 是否至少有 1 个点写到了“设计目标 + 实现机制 + tradeoff”。
- 是否已经识别出真正的抽象层，而不是把普通组件误判成亮点。
- 是否有足够的证据支撑最强亮点。
- 如果存在复杂链路，是否加了 Mermaid 或等价的流程表达。

## 质量要求

- 不要编造没有证据支撑的职责或结果。
- 尽量把结论绑定到具体文件、模块或配置。
- 多用具体模块、配置和流程，少用空泛赞美。
- 目标是面试可用性，不是 code review 完整性。

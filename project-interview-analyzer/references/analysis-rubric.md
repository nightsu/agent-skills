# 分析标准

用下面这些信号把一个仓库整理成面试材料。

## 优先关注什么

- 产品范围：项目解决什么问题、给谁用。
- 技术栈：框架、语言、构建工具、状态管理、数据层、样式、测试。
- 架构：路由、模块边界、业务分层、共享组件、服务层、monorepo 结构。
- 复杂度：登录鉴权、权限控制、SSR/CSR 取舍、性能、缓存、表单、表格、图表、上传、实时、国际化、离线支持。
- 工程质量：lint、测试、CI、发版流程、错误处理、可观测性、类型安全。
- 迁移工作：框架升级、接口变化、仓库重组、设计系统接入。

## 面试友好角度

- 哪些地方最难？
- 做了什么决策，这个决策背后的 tradeoff 是什么？
- 这个难点/亮点的设计目标是什么？是怎么实现的？为什么这样比常规做法更好？
- 你如何提升稳定性、性能或开发体验？
- 哪些部分是从零搭建，哪些是集成现成能力？
- 仓库里有哪些证据能支撑这段故事？

## 深挖优先级

- 优先深挖会影响多个页面、多个业务入口或多个系统边界的抽象层：认证/授权守卫、请求封装、数据适配层、分页/列表控制器、状态容器、路由/跳转封装、嵌入容器、消息/事件桥接层。
- 优先深挖有明显输入-处理-输出结构的链路：认证、购买、绑定、跳转、支付/提交、导出/导入、复杂表单、文件上传、回调处理、任务流转。
- 优先深挖能说明“设计决策”的点，而不是只描述“页面长什么样”。

## 亮点判定

以下内容才适合被写成“最强亮点”：

- 能被多个页面复用的抽象层
- 明确解决了一个跨页面、跨入口、跨设备或跨系统的问题
- 代码里能找到清晰的输入、处理、输出和副作用
- 有明显的设计选择，而不是只有“把功能写出来”

以下内容不应单独作为亮点：

- 单页静态展示
- 只做了样式调整
- 没有明显复用边界的小组件
- 不能说明设计决策的普通 API 调用

## 输出提醒

- 区分事实和推测。
- 优先给出具体文件或模块引用。
- 不确定的地方直接说明，不要用猜测填空。

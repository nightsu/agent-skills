# 通用检查清单

所有项目都先过这一轮，不分类型。

## 先回答这几个问题

- 项目解决什么问题？
- 谁在使用它，谁为结果负责？
- 仓库的主要形态是什么：业务应用、组件库，还是基建 / 平台？
- 哪些模块最复杂，复杂度来自哪里？

## 必扫维度

- 产品范围：项目做什么、边界在哪里。
- 技术栈：框架、语言、构建工具、状态管理、数据层、样式、测试。
- 架构：路由、模块边界、业务分层、共享组件、服务层、monorepo 结构。
- 复杂度：状态、异步、副作用、跨系统边界、动态结构、文件处理、实时性、权限、性能。
- 工程质量：lint、测试、CI、发版流程、错误处理、可观测性、类型安全。
- 演进信号：框架升级、接口变化、仓库重组、抽象收敛、历史兼容。

## 面试友好角度

- 哪些地方最难？
- 做了什么决策，这个决策背后的 tradeoff 是什么？
- 这个难点 / 亮点的设计目标是什么？是怎么实现的？为什么这样比常规做法更好？
- 你如何提升稳定性、性能、开发体验或一致性？
- 哪些部分是从零搭建，哪些是集成现成能力？
- 仓库里有哪些证据能支撑这段故事？

## 优先深挖什么

- 跨页面、跨包、跨系统复用的抽象层
- 有明显输入-处理-输出结构的链路
- 多个组件协同完成的复杂交互
- 设计上存在明显取舍的模块

## 不要误判成亮点的内容

- 单页静态展示
- 只做了样式调整
- 没有明显复用边界的小组件
- 不能说明设计决策的普通 API 调用

## 输出提醒

- 区分事实和推测。
- 优先给出具体文件或模块引用。
- 不确定的地方直接说明，不要用猜测填空。

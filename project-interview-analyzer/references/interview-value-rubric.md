# 面试价值 Rubric

这个 reference 用来做 `project-interview-analyzer` 的出口门禁。目标不是把项目写得更复杂，而是筛掉面试价值不足的内容，只保留能被追问、能讲设计判断、且有证据支撑的点。

## 核心原则

复杂边界只是候选信号，不自动等于面试价值。一个点只有能说明“为什么难”和“为什么值得讲”，才可能进入 `主讲`。

不要因为代码量多、页面视觉复杂、组件很多、调用了新技术，就直接判断为亮点。要先确认它是否改变了业务正确性、状态生命周期、协议模型、可靠性、扩展复用或性能体验取舍。

## 主讲门槛

一个候选点进入 `主讲`，原则上至少满足 4/6：

| 维度 | 成立条件 | 不成立的表现 |
| --- | --- | --- |
| 真实问题 | 解决业务、用户、工程或稳定性上的明确问题 | 只是“实现了某功能” |
| 设计判断 | 有取舍、边界划分、抽象或策略选择 | 只是按接口字段渲染 |
| 实现机制 | 能讲清输入、状态、处理、输出、副作用 | 只能说页面长什么样 |
| 复杂边界 | 命中项目形态对应的状态、协议、可靠性或扩展边界 | 只有局部组件逻辑 |
| 可追问 | 面试官能继续追问方案、失败场景、替代方案 | 追问后只能复述功能 |
| 证据支撑 | 有源码、配置、文档、调用链或测试证据 | 主要靠推测或个人包装 |

不满足 4/6 时，通常只能进入 `备选` 或 `剔除`。如果用户明确说明了个人职责、上线结果或业务指标，可以提高可信度；如果仓库中没有证据，必须标注 `待确认`。

## 分级定义

- `主讲`：可讲 3-5 分钟，能覆盖问题定义、设计目标、实现机制、tradeoff、证据和追问空间。适合进入“最强亮点”“最难的问题”“简历 bullet”。
- `备选`：能支撑主讲，或用于回答追问、补充架构背景，但不单独包装成最强亮点。
- `剔除`：不进入亮点，最多在功能范围里一笔带过。剔除不是否定工作量，而是避免材料膨胀。

## 通用价值维度

先按这些维度判断，再结合项目形态给例子：

- 业务正确性：是否影响钱、权限、推荐、履约、内容发布、用户决策或关键转化。
- 状态与生命周期：是否存在跨页面、跨端、跨系统、实时、离线、恢复、并发或长任务状态。
- 协议与模型：是否有领域建模、DTO/schema 转换、多版本兼容、数据一致性或权限边界。
- 可靠性与风险控制：是否涉及失败、重试、取消、回滚、幂等、审计、误操作保护或降级。
- 扩展性与复用：是否沉淀了可复用抽象、插件机制、配置模型、组件 API 或平台能力。
- 性能与体验：是否在性能、体验、准确性、稳定性、成本、可维护性之间做过明确取舍。

## 项目形态判断

下面不是封闭清单，而是候选信号。每个候选都必须继续回答“主讲成立条件”和“低价值反例”。

### 业务 / 产品应用

候选信号：

- 转化链路、交易/支付/履约、内容生产发布、个性化推荐、实验分流。
- 跨端同步、实时协作、关键决策工作流、复杂表单或编辑器、弱网/恢复、权限与合规。
- 用户操作会改变状态结构、数据结构、风险边界或关键路径。

主讲成立条件：

- 能讲清业务正确性要求，例如错误会影响推荐、交易、履约、发布、权限或用户决策。
- 有状态机、数据适配、异常恢复、误操作保护、权限边界或跨模块协同。
- 有明确取舍，例如配置化 vs 定制化、实时体验 vs 一致性、编辑效率 vs 错误风险。

低价值反例：

- 普通路由配置、普通埋点调用、普通筛选展示、普通 Modal。
- 单页静态展示、按接口字段渲染列表、只有样式或布局调整。

### 组件库 / 设计系统

候选信号：

- API 设计取舍、组合模型、受控/非受控兼容、token 与主题扩展。
- 可访问性、国际化、breaking change 迁移、测试矩阵、tree-shaking / bundle size。
- 跨框架、跨产品线或跨业务场景复用。

主讲成立条件：

- 能讲清使用者如何组合、扩展、迁移或避免误用。
- 有兼容策略、版本策略、测试覆盖、类型约束、主题机制或性能约束。
- 能说明为什么这个 API 不是只为一个页面服务。

低价值反例：

- 单个组件样式、普通 props 封装、demo 堆叠。
- 只把第三方组件换皮，没有 API、状态或兼容性设计。

### 基建 / 平台 / 工具

候选信号：

- 构建链路、插件机制、配置模型、缓存与增量策略、发布/回滚。
- 多环境隔离、迁移成本控制、可观测性、DX、错误提示、平台权限和资源治理。
- 影响多个项目、团队、环境或发布链路。

主讲成立条件：

- 能讲清输入配置、处理流程、输出产物、失败场景和回滚方式。
- 有扩展点、缓存策略、兼容策略、迁移策略或可观测性设计。
- 能说明如何降低使用成本、发布风险或维护成本。

低价值反例：

- 普通脚本拼接、单次配置调整、只升级依赖版本。
- 只把命令包装了一层，没有错误处理、扩展点或迁移价值。

### 后端服务 / 数据 / AI 应用

候选信号：

- 领域模型、状态机、幂等、事务/最终一致性、队列/任务/重试、权限审计。
- 数据质量、检索/RAG、prompt/tool 编排、模型降级、评测与可观测性、成本/延迟控制。
- 多系统协议转换、长任务、异步结果、审计或合规边界。

主讲成立条件：

- 能讲清请求入口、领域状态、处理链路、失败恢复、输出结果和副作用。
- 对 AI/RAG/prompt/tool 类项目，必须能说明评测、降级、可观测性、成本或延迟取舍中的至少一项。
- 对数据/后端类项目，必须能说明一致性、幂等、权限、事务、任务重试或审计中的至少一项。

低价值反例：

- 普通 CRUD、单次模型调用、无评估闭环的 prompt 包装。
- 只有接口转发，没有领域建模、协议适配、可靠性或可观测性设计。

## Review 输出格式

最终材料中建议保留一个简短小节：

```md
## 面试价值 review

### 主讲

- <亮点>：满足 <x>/6。证据：<文件/模块>。为什么值得讲：<一句话>。

### 备选

- <点>：适合作为 <主讲点> 的支撑或追问补充。

### 剔除

- <点>：原因是 <普通 UI / 单次 API / 纯样式 / 无设计判断 / 证据不足>。
```

如果项目很简单，`主讲` 可以少，但要说明原因。不要为了凑数量把 `备选` 或 `剔除` 改写成亮点。

# Document Analysis

对文档进行结构化理解、摘要和审阅。默认优先识别文档类型，再决定输出 summary 还是 review。

## 目标

- 把文档先分类型，再输出对应结果
- 对技术/需求/方案类文档，不只做摘要，还要找出逻辑冲突、缺失前提、风险点和待确认项
- 对一般文章，保留简洁摘要即可

## 输入

支持以下输入方式：

- 用户直接粘贴文本
- 用户提供 Markdown / 纯文本 / PDF 文件路径
- 用户提供 URL；如果当前环境能直接读取正文，则先解析正文再分析
- 其他技能传入文本内容

这些输入方式是平级的。`document-analysis` 负责分析已经拿到的正文；如果当前环境无法直接读取正文，要明确提示：

- 请粘贴正文
- 或提供导出的 Markdown / PDF
- 或先用专门的导入 skill 把链接内容转换成文本

如果输入是文件，先读取全文；如果文本过长，按章节或逻辑段落分块处理。

## 文档类型

先判断文档属于哪一类：

- `requirements`：需求文档、PRD、功能说明
- `technical`：技术设计、架构说明、实现复盘
- `proposal`：方案文档、评审材料、对比分析
- `general`：普通文章、博客、杂谈

如果无法稳定判断，先给出 `inferred` 类型并说明依据。

## 工作流

### 1. 识别类型

- 找出标题、章节、术语、表格、结论段
- 判断文档是偏业务、偏技术、偏方案，还是一般文章
- 标记不确定的地方，不要硬分类

### 2. 提炼事实

- 提取核心主题、目标、约束、依赖、结论
- 对技术/需求文档，额外提取前提条件、边界、验收口径、假设

### 3. 选择模式

- `summary`：面向一般文章，输出简洁结构化摘要
- `review`：面向需求/技术/方案文档，输出摘要 + 问题发现 + 优先级建议

### 4. 生成结果

#### summary 模式

- 结构化摘要
- 关键要点
- 如果用户需要，可以补一版更短或更详细的摘要

#### review 模式

除摘要外，必须输出：

- 逻辑不一致点
- 缺失信息 / 缺失前提
- 风险点
- 建议优先级
- 待确认事项

优先级建议使用三档：

- `P0`：会直接影响可行性、结论或交付
- `P1`：会影响实现方案、范围或测试
- `P2`：会影响表达完整性，但不阻塞当前判断

## 审阅规则

对技术/需求/方案文档，优先追问这些问题：

- 目标是否明确
- 范围是否闭合
- 关键前提是否缺失
- 方案和约束是否自洽
- 依赖和边界是否说清
- 是否有明显的风险或反例没覆盖

如果文档存在明显缺口，不要只说“内容不完整”，要明确指出：

- 缺什么
- 为什么重要
- 应该问谁
- 不补会造成什么影响

## 输出要求

输出尽量保持 Markdown，且包含：

- 文档类型
- 摘要
- 问题清单
- 优先级
- 待确认事项

对一般文章，只保留摘要和关键要点。

## 输出模板

按 `references/output-templates.md` 中的模板输出对应结构。

## 边界

- 不负责排版美化，格式问题交给 `$markdown-lint`
- 不负责环境配置引导，这类问题直接问用户或使用专门工具
- 不负责实现特定平台的鉴权和抓取细节；这类工作可由单独的导入/适配 skill 完成
- 不保留文学性输出；如果以后需要文学性，再单独做一个 skill

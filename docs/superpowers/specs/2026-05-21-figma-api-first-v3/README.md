> Status: Draft, awaiting user review

## TL;DR

第 3 版补齐 Phase C-up 的 `figma-api-first` skill,用来替代手填 `api-mapping.md`。

第一版只支持用户粘贴接口结构 / 返回值类型 / 字段清单,暂不接 YApi、Swagger 或 OpenAPI 平台抓取:

```text
用户粘贴接口结构
  → figma-api-first
  → docs/design/<feature>/api-mapping.md
  → figma-ui-api-mapper 消费 api-mapping.md 并结合 Figma 生成 component-mapping.md
```

`figma-api-first` 不属于 `figma-ui-api-mapper`。两者同属 Phase C,但职责不同:

- C-up `figma-api-first`:接口事实 → `api-mapping.md`
- C-low `figma-ui-api-mapper`:Figma UI + `api-mapping.md` → `component-mapping.md`

---

## Why v3

MVP 和 v2 已经验证:

- A/B 可以产出需求事实和 UI 理解。
- C-low/D/E 可以消费 `api-mapping.md` 并继续生成实现规格。
- 手填 C-up 是当前链路里最容易产生字段遗漏、类型误读和 UI 槽位数量误判的位置。

真实项目验证暴露过一个问题:如果只看截图填写 `api-mapping.md`,容易把接口字段数和 UI 槽位数混在一起判断。`figma-api-first` 的目标是先把接口事实稳定下来,再交给 C-low 与 Figma UI 做绑定。

---

## Scope

### In Scope

- 新增 `figma-api-first` skill,处于 Phase C-up。
- 读取 `docs/design/<feature>/clarified-requirement.md` 和 `ui-understanding.md` 作为语义辅助。
- 接收用户粘贴的接口返回结构、TypeScript type、JSON 示例或字段清单。
- 产出 `docs/design/<feature>/api-mapping.md`。
- 在 `docs/design/<feature>/inputs.md` 追加 audit 记录。
- 用 Open Questions 暴露缺失字段、类型不确定、分页/权限/错误态不确定等问题。
- 保持 `api-mapping.md` 兼容现有 `figma-workflow/templates/api-mapping.md`。

### Out of Scope

- 不接 YApi / Swagger / OpenAPI 自动抓取。
- 不调用 Figma MCP。
- 不读取 raw Figma JSON。
- 不做 UI 节点绑定,不产出 `component-mapping.md`。
- 不修改业务代码。
- 不生成 OpenSpec proposal。
- 不根据接口结构推断视觉布局。

## Position in Workflow

```text
phase A → phase B → phase C-up → phase C-low → phase D → phase E
clarify   UI理解     api-first    ui-api-mapper  design-token  emit-spec
```

进入条件:

- `docs/design/<feature>/clarified-requirement.md` 存在且非占位。
- `docs/design/<feature>/ui-understanding.md` 存在且非占位。
- 用户提供至少一种接口结构输入。

输出:

- `docs/design/<feature>/api-mapping.md`

下游:

- `figma-ui-api-mapper` 读取 `api-mapping.md`,再结合 Figma node 做 UI/API 绑定。

## Input Contract

`figma-api-first feature=<feature-name>`

必须输入:

- `feature=<feature-name>`
- 用户粘贴的接口结构,支持以下任一形式:
  - TypeScript interface / type。
  - JSON response 示例。
  - 字段清单表格或 bullet list。
  - 后端接口说明文本。

可选输入:

- API 名称或 URL。
- Method。
- 业务用途说明。
- 请求参数。
- 错误码或空态说明。

第一版不要求结构化 CLI 参数。用户可直接在对话里粘贴接口结构;skill 负责把它整理成 `api-mapping.md`。

---

## Output Contract

`api-mapping.md` 必须保持 Markdown 可读、git diff 友好,并兼容现有模板:

```markdown
# API Mapping — <feature>

> Generated by figma-api-first@0.1.0 at <ISO8601>
> Source: pasted schema

## Data Sources

| UI Module | API | Method | Description |
|---|---|---|---|
| <module or page> | <api name/url> | GET/POST/UNKNOWN | <business purpose> |

## Field Mapping

| UI Field | API Field | Type | Transform |
|---|---|---|---|
| <ui candidate or UNKNOWN> | data.foo.bar | number | percent / currency / count / raw |

## State Mapping

| State | Trigger | UI Behavior |
|---|---|---|
| loading | request pending | show loading placeholder |
| empty | response success but key list/count is empty | show empty state |
| error | request failed or success=false | show error state |

## Open Questions
- [ ] <unresolved api / type / pagination / permission / fallback question>
```

### Field Mapping Rules

- API 字段路径必须尽量完整,例如 `data.firstContactSection.callOutRate.percent`。
- Type 使用接口事实,允许值包括 `string` / `number` / `boolean` / `array` / `object` / `null` / `unknown`。
- Transform 只描述数据展示转换,例如 `percent`, `ratio`, `duration`, `currency`, `count`, `raw`。
- UI Field 可以来自 `ui-understanding.md` 的 module / suspected component / static-dynamic guess;无法稳定判断时写 `UNKNOWN` 并进入 Open Questions。
- 不因为接口字段存在就假设 UI 一定展示该字段。
- 不因为 UI 有槽位就伪造接口字段。

### State Mapping Rules

至少保留:

- `loading`
- `empty`
- `error`

如果接口结构包含 `code` / `success` / `msg`,需要映射:

- `success=false` 或 `code != 0` → error。
- 关键列表为空或关键指标缺失 → empty 或 partial-empty,不确定时写 Open Questions。

## Workflow

1. 解析 `feature=<feature-name>`;缺失则拒绝执行。
2. 确认 `docs/design/<feature>/` 存在;不存在时创建目录并提示确认 feature 名。
3. 读取 `clarified-requirement.md` 和 `ui-understanding.md`;缺失或仍是模板占位则拒绝进入 C-up。
4. 获取用户粘贴的接口结构;没有接口结构时请求用户提供。
5. 解析接口事实:
   - API 名称 / method / 业务用途。
   - response 根结构。
   - 可展示字段路径。
   - 类型。
   - 错误态 / 空态 / 分页 / 权限线索。
6. 对齐 A/B 产物中的业务 module 和 UI 候选槽位。
7. 生成 `api-mapping.md`。
8. 将无法确认的映射写入 `Open Questions`。
9. 追加 `inputs.md` audit 记录,包括 source type、api_first_version、feature、field_count、open_questions_count。
10. 输出 self-check,供 `figma-workflow` review gate 展示。

---

## Self-Check

| Check | Warning |
|---|---|
| `Data Sources` 为空 | "Data Sources 缺失,不能进入 C-low" |
| `Field Mapping` 为空 | "Field Mapping 缺失,不能进入 C-low" |
| 字段类型包含 `unknown` | "存在 unknown 类型,建议 review 后再继续" |
| `Open Questions` 存在非 deferred 项 | "存在未解决接口问题,建议 review 后再继续" |
| 仍含模板占位标记 | "产物仍含模板占位,orchestrator 会视为未完成" |

self-check 不阻塞产物落盘;是否继续由 `figma-workflow` 的 review gate 决定。

---

## Orchestrator Integration

`figma-workflow` 在 v3 中调整 C-up 行为:

- 如果缺 `api-mapping.md`,且 `figma-api-first` 可用,优先提示运行 `figma-api-first`。
- 如果 skill 不可用,或用户选择 fallback,继续提示手填 `templates/api-mapping.md`。
- 不自动从外部平台抓取接口。
- 不自动进入 C-low;C-up 完成后仍进入 review gate。

菜单建议:

```text
Next step:
  [1] Run figma-api-first (phase C-up)
  [2] Use template fallback
  [3] Manually edit a product
  [4] Exit
```

## Compatibility

- 现有手填 `api-mapping.md` 仍有效。
- `figma-ui-api-mapper` 不需要知道 `api-mapping.md` 来自手填还是 `figma-api-first`。
- `figma-emit-spec` 仍按原方式读取 `api-mapping.md`。
- `figma-api-first` 不改变 Phase E coding boundary:业务代码只能在用户明确确认执行 coding 后开始。

---

## Validation

P10 验收至少覆盖:

- 可从用户粘贴的 TypeScript response type 生成 `api-mapping.md`。
- 生成产物包含 Data Sources / Field Mapping / State Mapping / Open Questions。
- 销售工作台 fixture 覆盖嵌套指标结构,例如 `data.firstContactSection.callOutRate.percent`。
- 缺 A 或 B 时拒绝执行。
- 不调用 Figma MCP。
- 不修改业务代码目录。
- `figma-workflow` 的 C-up 菜单优先推荐 `figma-api-first`,并保留模板 fallback。

---

## Open Questions

- 第一版是否需要把用户粘贴的原始接口结构保存为 `docs/design/<feature>/api-source.md`。建议 P10 暂不新增该产物,只在 `inputs.md` 记录 source type 和摘要,避免扩大文件契约。
- 对多个接口的数据源,是否按一个 `api-mapping.md` 汇总。建议第一版汇总到同一文件,每个 API 一行 Data Source。
- 是否要支持请求参数映射。建议第一版只在 Open Questions 中记录,不新增 Request Mapping 表。

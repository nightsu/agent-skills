## §5.1 Review Gate(所有阶段通用)

**触发时机:** skill 执行完毕、产物落盘后,**立即**触发,在 orchestrator 进度面板之前。

### 形态(决策 7-B)

```
✓ Phase D 产物已落盘:
  - docs/design/referral-home/design-token-patch.md (4.2 KB)
  - inputs.md (+1 entry)

Self-check:
  ⚠ 3 个 token 标记为 INFERRED (line 24, 31, 47) —— 建议人工核对
  ⚠ 2 个 module 在 component-mapping.md 出现但本产物未抽 token (BannerCarousel, FooterBar)
  ℹ 1 个 Figma variable 关联了 2 个 module

Recommended next: review 产物 → 进入 phase E

Choose:
  [1] Proceed to phase E (figma-emit-spec)
  [2] Re-run phase D (will overwrite; git is source of truth, commit before re-running)
  [3] Pause for manual edit (return here when done, then choose 1)
  [4] Exit workflow
```

### 4 个选项的语义(决策 7a-③、7b-③)

| 选项 | 语义 |
|---|---|
| [1] Proceed | 进下一阶段(orchestrator 路由到对应 skill 或回到手填提示) |
| [2] Re-run | 直接调用本阶段 skill,产物覆盖。**不做备份**,提醒用户 "git is source of truth, commit before re-running" |
| [3] Pause | orchestrator 退出;用户编辑文件后,再次运行 `figma-workflow feature=X`,产物驱动会自动检测"已被编辑",回到当前 review gate |
| [4] Exit | orchestrator 退出 |

### 自查规则(各阶段)

> 各阶段的具体自查规则在对应 skill 契约文档中定义。此处汇总:

| 阶段 | 自查项 | 详见 |
|---|---|---|
| A (手填) | 套件不参与,跳过 self-check;orchestrator 只检查"非模板占位" | [§3.1](./03-orchestrator.md#31-阶段进入条件产物驱动推断) |
| B (手填) | 同上 | 同上 |
| C1 (`api-mapping.md`,手填) | 同上 | 同上 |
| C2 (`figma-ui-api-mapper`) | needs_confirmation / unknown / 低置信度 / Open Questions 数 | [§4a](./04a-ui-api-mapper.md#自查规则用于-5-review-gate) |
| D (`figma-design-token`) | INFERRED 标记 / 缺 token 的 module | [§4b](./04b-design-token.md#自查规则用于-5-review-gate) |
| E (`figma-emit-spec`) | Open Questions 总数 / 冲突 / Verification 未勾 / INFERRED 数 | [§4c](./04c-emit-spec.md#自查规则用于-5-review-gate) |

**自查不阻塞**,只是提示。用户可以无视所有警告直接进入下一阶段(强制权)。

### 自查的展示规则

| 级别 | 标记 | 触发条件 |
|---|---|---|
| ⚠ Warning | 黄色提示 | 用户**应该**关注但不必修复 |
| ℹ Info | 蓝色提示 | 仅作信息提示 |
| ❌ Error | 红色提示 | 跨产物冲突 / 致命问题(仅 §4c emit-spec 阶段会有) |

## §5.2 Handoff 出口(只在 `figma-emit-spec` 完成后触发)

**触发时机:** `figma-emit-spec` 的常规 review gate(§5.1)显示后,**紧接着**追加 handoff 菜单。
用户从 [1][2][3][4] 中如果选 [1] (Proceed) ,进入 handoff 菜单;选 [2][3][4] 则按 §5.1 处理。

### 形态(决策 13:4 个选项;OpenSpec stub 砍掉)

```
[Phase E review gate above ...]

Detected environment:
  ✓ superpowers (brainstorming, writing-plans, executing-plans, ...)

Previous handoff choice: superpowers (saved 2026-05-20). Use [P] to repeat.

Handoff to apply stage:
  [1] Builtin — generate task-breakdown.md (smallest, you implement manually)
  [2] superpowers:writing-plans (detected — recommended)
  [3] Manual — exit, I'll take implementation-spec.md elsewhere
  [4] Pause for manual edit / answer open questions first

Choose:
```

### 4 个选项的行为

| 选项 | 行为 | 副作用 |
|---|---|---|
| [1] Builtin | orchestrator 在 emit-spec skill 内部跑一个"task breakdown"子流程,产出 `task-breakdown.md`(从 `implementation-spec.md` 的 Modules 章节拆出 task 清单) | 写入 `.workflow-prefs.json`(首次)或覆盖(后续) |
| [2] superpowers | orchestrator 调用 `Skill("superpowers:writing-plans")`,把 `implementation-spec.md` 作为输入传入,后续由 superpowers 接管 | 写入 `.workflow-prefs.json` |
| [3] Manual | 退出 workflow,不做任何事 | **不写**偏好(用户明确选了"我自己来",不应记忆) |
| [4] Pause | 退出 workflow,用户编辑 `open-questions.md` 或 `implementation-spec.md` 后重新调用 `figma-workflow`,会回到此 gate | **不写**偏好 |

### `[P]` 快捷键

如果 `.workflow-prefs.json` 存在且 `handoff_after_emit` 是 builtin/superpowers/manual/pause 之一,
菜单顶部显示 `Previous handoff choice: ... Use [P] to repeat.`,用户输入 `P` 等价于选上次的选项。

> **注意:** 即使有上次记录,菜单**仍然弹出**。决策 7a-③ 的精神 —— 用户始终在驾驶座,
> 不做自动化重放。`[P]` 只是省一次输入。

### 检测 superpowers 是否安装

orchestrator 在打印菜单前检测 superpowers 是否可用:
- 如果可用:[2] 后面追加 `(detected — recommended)`
- 如果不可用:[2] 后面追加 `(not detected — install superpowers plugin first)`,**仍然显示**该选项,
  用户如果选了它,orchestrator 透传"未检测到 superpowers"的错误并退回菜单

> 检测方式:尝试调用 `Skill` 工具枚举可用 skills,看是否包含 `superpowers:writing-plans`。
> 具体检测 API 在实现时确认。

### `task-breakdown.md` 形态(handoff = builtin)

```markdown
# Task Breakdown — <feature>

> Generated by figma-emit-spec@0.1.0 (builtin handoff) at <ISO8601>
> Source: docs/design/<feature>/implementation-spec.md

## Tasks

### Task 1 — Setup
- [ ] Define types for API responses (see API Usage 段)
- [ ] Wrap API calls with hooks: useDiamondOverview / useOperationEntries / useHotGoods

### Task 2 — DiamondPreviewCard
- [ ] Check if existing `DiamondCard` component is reusable
- [ ] Implement `DiamondPreviewCard` (per Modules.1 in spec)
- [ ] Apply design tokens (borderRadius / padding / colors)
- [ ] Handle states: loading / empty / error

### Task 3 — OperationEntryGrid
- [ ] Implement grid layout (4 columns)
- [ ] Data-driven rendering for entry items
- [ ] Click → jumpType routing

### Task 4 — HotProductList
- [ ] Implement list with empty / loading / error states
- [ ] Apply design tokens
- [ ] Click → product detail or 兑吧 H5

### Task 5 — Integration & Verification
- [ ] Wire all modules into ReferralHomePage
- [ ] Run Verification Checklist (see spec)
- [ ] Answer/resolve all open questions
```

**拆分规则(MVP 极简):**
- 第一个 task 固定为 "Setup"(类型 + hooks)
- 中间 N 个 task 一对一对应 spec 中的 Modules
- 最后一个 task 固定为 "Integration & Verification"
- 不做更精细的依赖图、优先级、估时

## §5.3 不在 MVP 内的功能(明确划线)

- ❌ Review gate 的差量预览(不显示和上一次产物的 diff)
- ❌ 自查规则的项目级自定义(规则硬编码在 skill 内)
- ❌ 多 handoff 同时触发(只能选一个)
- ❌ 自动答 open questions(用户必须手工)
- ❌ handoff 偏好的"每个阶段独立记忆"(只记 emit-spec 出口的一次)
- ❌ OpenSpec stub 生成(决策 13 砍掉)
- ❌ 自动调用 superpowers 而不弹菜单
- ❌ task-breakdown 的更精细拆分(依赖图、估时等)

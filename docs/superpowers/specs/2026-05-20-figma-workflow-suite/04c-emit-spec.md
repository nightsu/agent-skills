## 职责

合并 A/B/C/D 所有产物,产出 `implementation-spec.md`(Agent 编码主输入)+
`open-questions.md`(跨阶段问题汇总),并在出口处提供 handoff 选择。

---

A + B + C(上 + 下) + D 全部 5 份产物存在且非占位:
- `clarified-requirement.md`
- `ui-understanding.md`
- `api-mapping.md`
- `component-mapping.md`
- `design-token-patch.md`

---

无新增输入(纯合成 skill)。

## 产物

| 路径 | 类型 | 说明 |
|---|---|---|
| `docs/design/<feature>/implementation-spec.md` | 主产物 | Agent apply 阶段唯一主输入 |
| `docs/design/<feature>/open-questions.md` | 主产物 | 跨阶段未决问题汇总 |
| `docs/design/<feature>/inputs.md` | 追加一条 | 含 inputs read 列表 + handoff choice |
| `docs/design/<feature>/.workflow-prefs.json` | (handoff 首次选择时)创建 | 记 handoff 选择 |
| `docs/design/<feature>/task-breakdown.md` | (handoff = builtin) | 从 spec 拆出的 task 清单 |

> handoff 走 superpowers / manual / pause 时,不产出 `task-breakdown.md`。
> handoff 详细见 [§5](./05-review-gate-and-handoff.md)。

## 产物格式(`open-questions.md`)

```markdown
# Open Questions — <feature>

聚合自所有阶段产物中标记的待确认项。

## From Phase A (clarified-requirement.md)
- [ ] 入口数量是否固定?
- [ ] 列表是否分页?

## From Phase B (ui-understanding.md)
- [ ] 空状态是否有单独设计?

## From Phase C (component-mapping.md, needs_confirmation: true)
- [ ] node `cover_image_123`: 来源不明(banner 接口 / 固定素材)
- [ ] node `tag_456`: 标签文案可能来自接口

## From Phase D (design-token-patch.md, INFERRED)
- [ ] OperationEntryGrid.columnCount: INFERRED from 4-instance grid,需确认是否固定

## Cross-Product Conflicts (auto-detected)
- [ ] `api-mapping.md` 声明字段 `user.diamond.history`,但 `component-mapping.md` 未绑定到任何 UI node
```

## 关键行为

1. **合成时不引入新事实**,只重组上游产物的事实
2. 任何新增/推断的内容必须显式标 `(INFERRED)`,或写入 `open-questions.md`
3. 自检 open-questions 数量,数量 > 0 时在 review gate 明确提示
   "建议先回答 open questions 再 handoff"
4. handoff 出口详细行为见 §5
5. 不修改上游产物
6. 跨产物冲突检测见下方"Cross-product conflicts 检测"

---

## Cross-product conflicts 检测(MVP 最小集)

| 冲突 | 检测方式 |
|---|---|
| `api-mapping.md` 中的字段未在 `component-mapping.md` 中被绑定 | 字段名字符串匹配 |
| `component-mapping.md` 中的 module 未在 `design-token-patch.md` 中抽 token | module 名字符串匹配 |
| `ui-understanding.md` 中的 module 未在 `component-mapping.md` 中出现 | module 名字符串匹配 |

冲突一律写入 `open-questions.md` 的 `## Cross-Product Conflicts (auto-detected)` 段,
在 review gate 标红提示。

> ⚠ MVP 检测是粗粒度字符串匹配,可能 false positive。
> 用户可以直接在 `open-questions.md` 中标 `[x]` 表示已确认非冲突,
> orchestrator 不重写该段。

## 不做的事

- ❌ 不调用 Figma MCP
- ❌ 不联网
- ❌ 在 spec 里**绝对不引用** raw Figma JSON
- ❌ 不修改上游产物
- ❌ 不自动答 open questions
- ❌ 不在 handoff = manual / pause 时产出 `task-breakdown.md`

---

## 错误处理

| 情况 | 处理 |
|---|---|
| 任一上游产物缺失 | orchestrator 已拦截 |
| 上游产物互相冲突 | 写入 open-questions,review gate 标红;**不阻塞**产物落盘 |
| 上游产物 markdown 解析失败(格式严重崩坏) | 透传错误,产物**不落盘** |

---

## 自查规则(用于 §5 Review Gate)

| 项 | 阈值 | 提示文案 |
|---|---|---|
| Open Questions 总数 | > 0 | "N 个 open question 未解决 —— 强烈建议先回答再 handoff" |
| Cross-Product Conflicts 数 | > 0 | "N 个跨产物冲突 —— 见 open-questions.md ## Cross-Product Conflicts" |
| Verification Checklist 未勾选项 | > 0 | "Verification checklist 有 N 项待勾选" |
| INFERRED 标记数 | > 0 | "spec 中有 N 处 INFERRED 内容,建议人工核对" |

# Project Interview Value Review Skill Update - Design

## 背景

`project-interview-analyzer` 已经能广扫项目、识别项目形态、深挖复杂模块，并生成技术/业务介绍与面试讲述文档。但“面试价值”目前仍偏经验判断，容易出现两个问题：

- 把代码量多、页面复杂、视觉效果多的内容误包装成亮点。
- 对不同项目形态使用同一套复杂度清单，导致前端业务应用偏重，而组件库、基建、后端/AI 项目的价值判断不够准确。

本次更新目标是把“面试价值 review”收敛为 `project-interview-analyzer` 的强制出口门禁，而不是新建独立 skill。

## 目标

- 在 `project-interview-analyzer` 中加入强制的“面试价值 Review”阶段。
- 将面试价值具象化为可执行 rubric。
- 按项目形态解释“复杂边界”，避免一套前端交互清单套所有项目。
- 输出时必须区分 `主讲`、`备选`、`剔除`。
- 防止普通 UI、单次 API、纯样式、无设计判断的小组件进入主讲亮点。

## 非目标

- 不新建 `interview-value-review` skill。
- 不改变 `project-interview-analyzer` 的基础输出文件名。
- 不要求每个项目都必须写很多亮点；简单项目可以主讲点少，但必须说明原因。
- 不把仓库观察直接改写成个人职责或业绩结果。

## 设计

### 1. 更新 `SKILL.md`

在 `project-interview-analyzer/SKILL.md` 中新增或调整三个位置：

1. 在“判断项目形态”之后，加入“按项目形态解释复杂边界”的要求。
2. 在“深挖要求”或“复杂交互补扫要求”之后，加入“面试价值 Review”要求。
3. 在“产出检查”中加入面试价值门禁检查。

`SKILL.md` 只保留核心流程和强制要求，详细 rubric 放入 reference，避免主文件过长。

### Skill 设计约束

这次是更新既有 skill，不新建新 skill。实现时遵守：

- 不改 `project-interview-analyzer` 的 name / description，除非后续发现触发条件不够准确。
- `SKILL.md` 只写强制流程、何时读取 reference、必须输出什么；不要把完整 rubric 塞进主文件。
- 新 reference 保持单层引用：`SKILL.md` 直接链接 `references/interview-value-rubric.md`。
- reference 要写成可执行门禁，不写抽象口号；每条标准都要说明“什么时候算高价值、什么时候只是候选信号”。
- 不新增空目录、README 或额外说明文件。
- 不把 `using-superpowers` 的流程复制进该 skill；只吸收“出口 review 门禁”的思想。

### 2. 新增 reference

新增：

`project-interview-analyzer/references/interview-value-rubric.md`

内容包括：

- 通用面试价值门槛：
  - 真实问题
  - 设计判断
  - 实现机制
  - 复杂边界
  - 可追问
  - 证据支撑
- 主讲 / 备选 / 剔除定义。
- 按项目形态拆分复杂边界：
  - 业务应用
  - 组件库 / 设计系统
  - 基建 / 平台 / 工具
  - 后端服务 / 数据 / AI 应用
- 低价值反例。
- 最终输出建议格式。

### 3. 面试价值门槛

一个点进入 `主讲`，原则上至少满足 4/6：

- 有真实业务或工程问题。
- 有明确设计判断或取舍。
- 能讲清输入、状态、处理、输出、副作用。
- 存在项目形态对应的复杂边界。
- 能经受面试追问。
- 有源码、配置、文档或调用链证据。

不满足时只能进入 `备选` 或 `剔除`。

### 4. 价值维度与项目形态化复杂边界

复杂边界只是候选信号，不自动等于面试价值。某个点只有在能说清“它改变了设计、状态、协议、可靠性、扩展性或业务正确性”时，才可以进入 `主讲`。

reference 不应写成某类项目的常见实现清单，而应先按价值维度判断，再按项目形态给例子。

通用价值维度适用于所有项目：

- 业务正确性：是否影响钱、权限、推荐、履约、内容发布、用户决策或关键转化。
- 状态与生命周期：是否存在跨页面、跨端、跨系统、实时、离线、恢复、并发或长任务状态。
- 协议与模型：是否有领域建模、DTO/schema 转换、多版本兼容、数据一致性或权限边界。
- 可靠性与风险控制：是否涉及失败、重试、取消、回滚、幂等、审计、误操作保护或降级。
- 扩展性与复用：是否沉淀了可复用抽象、插件机制、配置模型、组件 API 或平台能力。
- 性能与体验：是否在性能、体验、准确性、稳定性、成本、可维护性之间做过明确取舍。

不同形态的例子只作为候选方向，不是封闭清单。每个候选都必须绑定“为什么难”和“为什么值得讲”：

- 业务 / 产品应用：候选可以来自转化链路、交易/支付/履约、内容生产发布、个性化推荐、实验分流、跨端同步、实时协作、关键决策工作流、复杂表单或编辑器、弱网/恢复、权限与合规。普通路由配置、普通埋点调用、普通筛选展示、普通 Modal 不应作为主讲点。
- 组件库 / 设计系统：候选可以来自 API 设计取舍、组合模型、受控/非受控兼容、token 与主题扩展、可访问性、国际化、breaking change 迁移、测试矩阵、tree-shaking / bundle size、跨框架或跨产品复用。单个组件样式、普通 props 封装、demo 堆叠不应作为主讲点。
- 基建 / 平台 / 工具：候选可以来自构建链路、插件机制、配置模型、缓存与增量策略、发布/回滚、多环境隔离、迁移成本控制、可观测性、DX、错误提示、平台权限和资源治理。普通脚本拼接、单次配置调整、只升级依赖版本不应作为主讲点。
- 后端服务 / 数据 / AI 应用：候选可以来自领域模型、状态机、幂等、事务/最终一致性、队列/任务/重试、权限审计、数据质量、检索/RAG、prompt/tool 编排、模型降级、评测与可观测性、成本/延迟控制。普通 CRUD、单次模型调用、无评估闭环的 prompt 包装不应作为主讲点。

reference 中要把这些写成“候选信号 -> 主讲成立条件 -> 低价值反例”的结构，避免只列技术名词。

### 5. 输出要求

最终材料应包含或体现：

- `主讲`：可讲 3-5 分钟，有设计目标、实现机制、tradeoff、证据和追问空间。
- `备选`：可用于支撑主讲或回答追问，但不单独包装成最强亮点。
- `剔除`：明确哪些内容不进入亮点，避免材料膨胀。

对于足够复杂的项目，建议保留一个 `面试价值 review` 小节。

## 验证计划

修改完成后执行：

- 检查 `SKILL.md` 中能找到 `interview-value-rubric.md` 引用。
- 检查 reference 文件包含 `主讲`、`备选`、`剔除`、`4/6`、各项目形态复杂边界。
- 检查 reference 文件对每类复杂边界都包含低价值反例，而不是只列技术名词。
- 检查 `SKILL.md` 保持精炼，没有把完整 rubric 复制进主文件。
- 检查 `project-interview-analyzer` 的产出检查包含面试价值门禁。
- 手动用 `zhiyuan-agent` 案例对照：
  - SSE 对话状态机应进入主讲。
  - 志愿表编辑器应进入主讲。
  - 请求/session/版本兼容可进入备选或辅助亮点。
  - 规划师卡片、普通视频播放、Toast/Modal/Network 应进入剔除或不主讲。

## 风险

- Rubric 写得过细会增加上下文负担，所以详细内容放 reference，只有在项目分析时按需读取。
- 如果只写“面试价值”而不写反例，仍可能回到主观判断，因此 reference 必须包含低价值反例。
- 如果只在 AGENTS.md 约束，本 skill 在其他仓库不会生效，所以应更新 skill 本体。

## TL;DR

P13 基于 P12 `.figma-cache/` 的 before / after evidence 产出:

```text
docs/design/<feature>/design-diff.md
```

`design-diff.md` 用来回答三个问题:

1. Figma 改了什么。
2. 哪些 Phase A-E 产物可能受影响。
3. 用户应该重跑哪些阶段,以及哪些变化可以只人工确认。

第一版只支持 **同一 file key + 同一 node id** 的两次快照对比,不支持跨 node 对比,不做像素级视觉 diff,不自动覆盖任何上游 `.md` 产物。

---

## Why P13

P12 解决了 Figma evidence 可缓存的问题,但缓存本身不告诉用户"设计改稿影响了什么"。真实工作流里常见的问题是:

- 设计师改了文案,但实现规格仍引用旧文案。
- Frame 结构或重复项数量变了,但 C2/D 没有重跑。
- 只改了颜色/圆角/字号,却误以为要从 A 到 E 全量重跑。
- Figma 改动发生在 Phase E 之后,用户不知道是否需要重新生成 `implementation-spec.md`。

P13 的目标是把改稿影响显式写成 reviewable Markdown,让用户在进入 coding 前做一次低成本判断。

---

## Scope

### In Scope

- 基于 `.figma-cache/` 的两个 snapshot 产出 `design-diff.md`。
- 第一版只对比同一 `file_key` + `node_id`。
- 对比 metadata、design context 和 screenshot metadata 中的可结构化字段。
- 输出 changed / added / removed nodes。
- 输出 text changes、size / layout changes、visual token signals、asset signals。
- 输出 likely affected modules。
- 输出 recommended rerun phases。
- 在 `inputs.md` 追加 diff audit 记录。
- 在 `figma-workflow` 中提示可生成或查看 `design-diff.md`。

### Out of Scope

- 不做跨 node diff。
- 不做像素级 screenshot diff。
- 不生成或修改 `ui-understanding.md` / `component-mapping.md` / `design-token-patch.md` / `implementation-spec.md`。
- 不自动重跑任何 Phase。
- 不把 diff 当作 review gate 的替代品。
- 不修改 Figma 文件。
- 不写业务代码。
- 不下载真实图片资源;资源处理留给 P15。

## Snapshot Model

P13 在 P12 cache 基础上增加 snapshot 约定,避免 refresh 覆盖掉 before evidence。

```text
docs/design/<feature>/.figma-cache/
├── manifest.json
├── metadata.<file-key>.<node-id-safe>.json
├── design-context.<file-key>.<node-id-safe>.json
├── screenshots/
│   └── screenshot.<file-key>.<node-id-safe>.json
└── snapshots/
    ├── 2026-05-21T10-05-00+08-00/
    │   ├── metadata.<file-key>.<node-id-safe>.json
    │   ├── design-context.<file-key>.<node-id-safe>.json
    │   └── screenshots/
    │       └── screenshot.<file-key>.<node-id-safe>.json
    └── 2026-05-21T11-20-00+08-00/
        ├── metadata.<file-key>.<node-id-safe>.json
        ├── design-context.<file-key>.<node-id-safe>.json
        └── screenshots/
            └── screenshot.<file-key>.<node-id-safe>.json
```

Snapshot id 规则:

- 使用 ISO8601 时间,但文件名中把 `:` 替换为 `-`。
- 每次 refresh 前,如果当前 entry 是 fresh,先保存为 baseline snapshot。
- 新读取的 evidence 可以保存为 latest snapshot,再更新 root cache entry。
- 如果没有 baseline snapshot,只创建 snapshot 并提示"暂无可对比基线"。

## Input Contract

建议调用方式:

```text
figma-design-diff feature=<feature-name> fileKey=<file-key> nodeId=<node-id>
```

必须输入:

- `feature=<feature-name>`
- `fileKey=<file-key>`
- `nodeId=<node-id>`
- 至少一个 baseline snapshot 或可从 git / root cache 保存出的 before evidence
- 一次 current evidence,来自 fresh cache 或重新读取 Figma MCP

可选输入:

- `baseline=<snapshot-id>`
- `current=<snapshot-id>`
- `refresh=true`,表示先重新读取 Figma MCP 再 diff

第一版默认:

- `baseline` 选择同一 file/node 最近一次 snapshot。
- `current` 选择当前 root cache entry 或 refresh 后的新 snapshot。
- 如果 baseline 和 current 的 `file_key` 或 `node_id` 不一致,拒绝生成 diff。

---

## Output Contract

`design-diff.md` 必须是 Markdown,便于 review 和 git diff。

```markdown
# Design Diff — <feature>

> Generated by figma-design-diff@0.1.0 at <ISO8601>
> Figma: <file-key> / <node-id>
> Baseline: <snapshot-id>
> Current: <snapshot-id>

## Summary

| Severity | Count | Notes |
|---|---:|---|
| high | 0 | No blocking structural changes |
| medium | 2 | Text and layout changed |
| low | 1 | Screenshot metadata changed |

## Compared Snapshots

| Snapshot | Captured At | Source | Content Hash |
|---|---|---|---|
| baseline | <time> | .figma-cache/snapshots/<id> | sha256:<hash> |
| current | <time> | .figma-cache/snapshots/<id> | sha256:<hash> |

## Changed Nodes

| Node | Change Type | Before | After | Impact |
|---|---|---|---|---|
| <node name/id> | text_changed | <old text> | <new text> | Phase B / E may need rerun |

## Added Nodes

| Node | Type | Parent | Likely Module |
|---|---|---|---|

## Removed Nodes

| Node | Type | Previous Parent | Likely Module |
|---|---|---|---|

## Text Changes

| Node | Before | After | Recommended Action |
|---|---|---|---|

## Layout Changes

| Node | Property | Before | After | Recommended Action |
|---|---|---|---|---|

## Visual Token Signals

| Token Type | Node | Before | After | Recommended Action |
|---|---|---|---|---|

## Asset Signals

| Asset | Change | Recommended Action |
|---|---|---|

## Recommended Rerun Phases

| Phase | Recommendation | Reason |
|---|---|---|
| B | review | Text changed |
| C2 | skip | No UI/API binding change detected |
| D | rerun | Color token changed |
| E | rerun | Downstream implementation spec may be stale |

## Open Questions

- [ ] <question that must be answered before coding>
```

## Diff Categories

第一版只做结构化 diff,不解释视觉意图。

| Category | Evidence | Examples |
|---|---|---|
| `node_added` | metadata children | 新增指标卡片、新增按钮 |
| `node_removed` | metadata children | 删除 tab、删除空态文案 |
| `text_changed` | text node / design context | 文案、标题、单位变化 |
| `layout_changed` | bounds / auto-layout summary | width、height、x/y、gap、direction |
| `token_changed` | fills / typography / radius / spacing summary | 颜色、字号、字重、圆角 |
| `asset_changed` | asset refs / screenshot metadata | 图片引用、icon 引用变化 |
| `unknown_changed` | evidence changed but category unknown | raw evidence hash 变化但无法归类 |

Severity 建议:

| Severity | Rule |
|---|---|
| `high` | 节点新增/删除、关键结构变化、可能影响数据绑定 |
| `medium` | 文案、布局、token 变化,需要用户 review 或重跑局部阶段 |
| `low` | screenshot metadata、非关键尺寸、无法确认影响的小变化 |

## Rerun Recommendation Rules

`design-diff.md` 只给建议,不自动重跑。

| Change | Recommended Phase |
|---|---|
| 新增/删除业务模块或重复项 | Phase B + C2 + D + E |
| 文案变化但结构未变 | Phase B review + E |
| UI/API 绑定候选变化 | C2 + E |
| 颜色、字号、圆角、间距变化 | D + E |
| 资源引用变化 | D review + P15 assets |
| 只有 screenshot metadata 变化 | Manual review |
| raw hash changed but无法归类 | Manual review,必要时重跑 B/D |

如果多个规则命中,取更严格的推荐。例如新增节点同时有 token 变化时,推荐 B/C2/D/E。

## Orchestrator Integration

`figma-workflow` 在 P13 后可以新增可选提示:

```text
Design diff:
  [ ] design-diff.md (not generated)

Diff actions:
  [D] Generate design-diff.md from latest cache snapshots
  [V] View existing design-diff.md
```

触发条件:

- `.figma-cache/manifest.json` 存在。
- 同一 file/node 至少有 baseline 和 current evidence。
- 或用户选择 refresh,允许先读取 current evidence 再生成 diff。

限制:

- diff action 不标记任何 Phase 完成。
- diff action 不自动覆盖 A/B/C/D/E 产物。
- diff action 不替代 Phase review gate。

---

## Compatibility

- 没有 `.figma-cache/` 时,提示先执行 P12 cache refresh。
- 只有一个 snapshot 时,可以写 `design-diff.md`,但 Summary 必须说明"暂无 before/after 差异,仅记录 baseline"。
- 没有 `design-diff.md` 时,现有 Phase A-E 流程不受影响。
- 旧 fixture 不需要强制补 diff。

---

## Validation

P13 实现完成后至少验证:

- 同一 file/node 的两个 snapshot 可以生成 `design-diff.md`。
- file/node 不一致时拒绝生成 diff。
- 节点新增/删除能进入 Added / Removed Nodes。
- 文案变化能进入 Text Changes。
- bounds 或 layout summary 变化能进入 Layout Changes。
- token summary 变化能进入 Visual Token Signals。
- `Recommended Rerun Phases` 能根据变化类型给出 B/C2/D/E 建议。
- diff 生成不修改 A/B/C/D/E 产物。
- diff 生成不写业务代码。

## Open Questions

- 是否需要从 git 历史读取 before cache。建议第一版不做,只用 `.figma-cache/snapshots/`。
- 是否需要像素级 screenshot diff。建议放到 P15 automated validation 或更后续版本。
- `figma-design-diff` 是否作为独立 skill 暴露。建议第一版作为 v4 工程化 skill 暴露,同时由 `figma-workflow` 提供入口。

## TL;DR

P14 新增 `ui-handoff.md`,给设计师 / 产品一份最低交接规范和当前 feature 的上游输入质量清单。

```text
docs/design/<feature>/ui-handoff.md
```

它解决的是"Figma 和需求输入要怎样补齐,才能减少 B/C-low/D/E 的 unknown 和返工"。

注意:这里的 **UI handoff** 是面向设计/产品的上游交接文档,不是 Phase E 的 implementation handoff,也不是 coding 入口。

---

## Why P14

前面的阶段已经能从 Figma 和接口结构生成实现规格,但真实项目里仍会出现一些上游质量问题:

- text `characters` 缺失或与 `node.name` 不一致。
- Frame / Section 边界不清,导致 UI module 拆分不稳定。
- 重复项没有明确 template 与 instance 关系。
- loading / empty / error / permission 等状态没有覆盖。
- 图片、icon、插画等资源没有标注来源或交付方式。
- `figma-ui-understand` / `figma-ui-api-mapper` 产物里出现较多 `unknown`。

P14 的目标是把这些问题整理成设计/产品可读的检查单,帮助用户回到上游修正输入,而不是把不确定性一路带进 coding。

---

## Scope

### In Scope

- 新增 `figma-ui-handoff` skill。
- 读取 `docs/design/<feature>/` 下已存在的上游产物。
- 可选读取 `design-diff.md`,把改稿影响写入 handoff。
- 汇总 Figma selection、text、命名、重复项、状态、资源和 known gaps。
- 输出 `docs/design/<feature>/ui-handoff.md`。
- 在 `docs/design/<feature>/inputs.md` 追加 audit 记录。
- 在 `figma-workflow` 中提供可选入口。

### Out of Scope

- 不修改 Figma 文件。
- 不强制设计师使用特定设计系统。
- 不生成 implementation spec。
- 不替代 `figma-emit-spec`。
- 不自动重跑 A/B/C/D/E。
- 不写业务代码。
- 不下载真实资源;资源下载和自动化验证交给 P15。

---

## Position in Workflow

`ui-handoff.md` 是辅助产物,不属于 Phase A-E 的必需输入。

建议生成时机:

```text
Phase B 之后: 可生成初版,帮助设计/产品补齐 Figma 结构和文案
Phase C-low 之后: 可补充 unknown / API binding gaps
Phase D 之后: 可补充 token / asset gaps
P13 diff 之后: 可补充改稿影响和需要设计确认的变化
Phase E 之前: 推荐查看,但不强制阻塞
```

`figma-workflow` 可以在进度面板中显示:

```text
UI handoff:
  [ ] ui-handoff.md (not generated)
```

但它不能因为 `ui-handoff.md` 缺失就阻塞 Phase E。

## Input Contract

建议调用方式:

```text
figma-ui-handoff feature=<feature-name>
```

必须输入:

- `feature=<feature-name>`
- `docs/design/<feature>/` 存在

建议读取:

- `clarified-requirement.md`
- `ui-understanding.md`
- `component-mapping.md`
- `design-token-patch.md`
- `design-diff.md`(可选)
- `open-questions.md`(可选)

第一版规则:

- 缺少某个上游产物时仍可生成,但必须在 `Known Gaps` 中说明。
- 不把缺失产物写成已确认事实。
- 不复制 raw Figma JSON。
- 不把 `node.name` 当作真实文案事实来源。

## Handoff Rules

### Required Figma Selection

必须说明当前选区是否足够支持实现规格:

- 是否选择了完整页面 / frame,而不是局部卡片。
- 是否包含 modal、drawer、tooltip、popover 等浮层。
- 是否包含 responsive 变体或至少说明适配策略。
- 是否有外部依赖页面需要另行提供。

### Text Requirements

必须强调真实文案来自 text `characters`,不是 `node.name`:

- 文案不能只存在于 layer name。
- placeholder 文案需要标记为 placeholder。
- 可配置文案需要标注来源。
- 长文案、空态文案、错误文案要单独列出。

### Component / Section Naming Rules

命名建议只做最低规范:

- 页面主 frame 名能表达业务页面。
- Section / Frame 名能表达业务区域。
- 重复项 template 有稳定名称。
- 状态 frame 使用统一后缀,例如 `/loading`、`/empty`、`/error`。

不要求特定设计系统命名法,不强制使用某个组件库。

### Repeat Group Hints

必须把重复结构拆成 template 与 instance:

- 指标卡、列表项、表格行、tab、filter item 等需要标出重复模式。
- instance 数量变化是否代表真实数据量,需要产品确认。
- 如果 UI 槽位数和 API 字段数不一致,必须进入 `Known Gaps`。

### State Coverage

最低状态覆盖:

- loading
- empty
- error
- permission / no access
- partial data / degraded data(如适用)

如果状态未在 Figma 中出现,写为 `missing`,不能推断成已覆盖。

### Asset Marking

资产标注只写交接要求,不下载资源:

- 图片 / icon / 插画是否需要导出。
- 是否为占位图。
- 是否来自设计系统、远程 URL、业务接口或静态资源。
- 是否需要透明背景、倍率、格式或裁剪规则。

## Relationship to Existing Products

| Product | Used For |
|---|---|
| `clarified-requirement.md` | 判断业务目标、约束和待确认问题 |
| `ui-understanding.md` | 提取页面结构、重复项、unknown |
| `component-mapping.md` | 提取 API/UI 绑定 unknown、静态/动态归因问题 |
| `design-token-patch.md` | 提取 token 和 asset gaps |
| `design-diff.md` | 提取改稿影响和需要设计确认的变化 |
| `open-questions.md` | 汇总跨阶段未决问题 |

`ui-handoff.md` 只能引用这些产物中已经出现的证据或明确标记为需要确认的问题。

## Orchestrator Integration

`figma-workflow` 可以新增可选菜单:

```text
UI handoff:
  [ ] ui-handoff.md (not generated)

Handoff actions:
  [U] Generate ui-handoff.md
  [V] View existing ui-handoff.md
```

触发建议:

- Phase B 之后即可生成。
- Phase C-low/D/P13 之后生成的信息更完整。
- Phase E 前提示用户查看,但不强制。

限制:

- 不把 `ui-handoff.md` 纳入 Phase A-E 完成判断。
- 不因为 handoff gaps 自动阻塞。
- 不自动修改 Figma 或上游产物。

---

## Compatibility

- 旧 feature 没有 `ui-handoff.md` 时,现有流程不受影响。
- 没有 P13 `design-diff.md` 时,`Design Diff Follow-up` 可以为空并说明未生成。
- 没有 Phase E `open-questions.md` 时,只汇总已有阶段的 Open Questions。
- P14 不改变 Phase E implementation handoff 的语义。

---

## Validation

P14 实现完成后至少验证:

- 能在缺少 `design-diff.md` 时生成 `ui-handoff.md`。
- 能从 `ui-understanding.md` 收集 unknown / repeated patterns。
- 能从 `component-mapping.md` 收集 API/UI 绑定 gaps。
- 能从 `design-token-patch.md` 收集 token / asset gaps。
- 能从 `design-diff.md` 收集改稿 follow-up。
- `ui-handoff.md` 包含 Required Figma Selection、Text Requirements、Component / Section Naming Rules、Repeat Group Hints、State Coverage、Asset Marking、Known Gaps。
- 不修改 A/B/C/D/E 产物。
- 不写业务代码。

---

## Open Questions

- `ui-handoff.md` 是否应该在 Phase B 后自动建议生成。建议第一版只提示,由用户选择。
- 是否需要区分设计师版和产品版。建议第一版合并成一份轻量文档。
- 是否需要把 `ui-handoff.md` 作为 Phase E 输入。建议第一版不作为必需输入,只作为辅助证据。

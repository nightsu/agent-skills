## TL;DR

P15 为 figma-workflow-suite 补最后一层工程化收口:

```text
docs/design/<feature>/assets-manifest.md
docs/design/<feature>/validation-report.md
```

`assets-manifest.md` 规范图片 / icon / 插画等资源如何被引用、是否需要下载、下载到哪里、何时可以延后。

`validation-report.md` 记录最低成本的自动化验证结果,先覆盖 Markdown contract / fixture contract / boundary check。LLM-as-judge 只作为可选风险提示,不能作为唯一验收。

P15 不默认下载所有资源,不写业务代码,不改变 Phase E 后的 coding boundary。

---

## Why P15

P12-P14 已经补齐 cache、diff 和 UI handoff,但仍有两个收口问题:

- 资源引用散落在 `design-token-patch.md`、`ui-handoff.md`、cache evidence 或 Figma context 中,后续实现时容易漏掉图片/icon/插画。
- suite 的验证仍以人工 review fixture 为主,缺少最低成本的结构化检查,长期容易出现模板段落缺失、边界文案回退、fixture 漂移。

P15 的目标是把资源交付和验证从"口头约定"变成可 review 的文档与脚本约定。

---

## Scope

### In Scope

- 新增 `figma-assets-validate` skill。
- 读取 `design-token-patch.md`、`ui-handoff.md`、`design-diff.md` 和 `.figma-cache/` 中的资源引用。
- 产出 `docs/design/<feature>/assets-manifest.md`。
- 可选下载资源到 `docs/design/<feature>/assets/`,但第一版默认只记录引用和下载计划。
- 产出 `docs/design/<feature>/validation-report.md`。
- 提供 Markdown contract check,验证关键产物的必需章节和边界文案。
- 提供 fixture contract check,验证 suite / per-skill fixture 是否包含 README、inputs、expected 或明确说明。
- 提供 boundary check,确认 P12-P15 不写业务代码、不把 raw Figma JSON 写进 implementation spec。
- 可选 LLM-as-judge,只用于文档等价和风险提示。

### Out of Scope

- 不默认下载所有图片、icon 或截图。
- 不提交大体积二进制资源作为默认行为。
- 不做像素级视觉回归测试。
- 不把 LLM-as-judge 当作唯一验收。
- 不替代人工 review gate。
- 不修改 Figma 文件。
- 不自动重跑 A/B/C/D/E。
- 不写业务代码。

## Asset Manifest Contract

建议产物:

```text
docs/design/<feature>/assets-manifest.md
```

第一版使用 Markdown,便于 review。后续如需脚本消费,可以从同一信息生成 `assets-manifest.json`。

```markdown
# Assets Manifest — <feature>

> Generated by figma-assets-validate@0.1.0 at <ISO8601>
> Source: design-token-patch.md / ui-handoff.md / design-diff.md / .figma-cache

## Summary

| Type | Count | Download Required | Deferred |
|---|---:|---:|---:|
| image | 0 | 0 | 0 |
| icon | 0 | 0 | 0 |
| illustration | 0 | 0 | 0 |
| screenshot | 0 | 0 | 0 |

## Asset References

| Asset ID | Type | Source | Figma Node | Recommended Format | Required | Destination | Status | Notes |
|---|---|---|---|---|---|---|---|---|
| icon-filter | icon | design-token-patch.md | 123:456 | svg | yes | assets/icon-filter.svg | pending | Used by filter button |

## Download Plan

| Asset ID | Action | Command / Tool | Blocking |
|---|---|---|---|
| icon-filter | export from Figma | figma export or manual design handoff | yes |

## Deferred Assets

| Asset ID | Reason | Owner | Follow-up |
|---|---|---|---|

## Open Questions

- [ ] <asset question that must be resolved before coding if blocking>
```

字段规则:

| Field | Rule |
|---|---|
| `Asset ID` | 稳定 kebab-case 名称,不使用随机 hash |
| `Type` | `image` / `icon` / `illustration` / `screenshot` / `background` / `unknown` |
| `Source` | 资源引用来自哪个上游产物 |
| `Figma Node` | 有 node id 时填写,否则写 `unknown` |
| `Recommended Format` | `svg` / `png` / `webp` / `jpg` / `skip` / `unknown` |
| `Required` | `yes` / `no` / `deferred` |
| `Destination` | 推荐相对路径,例如 `assets/icon-filter.svg` |
| `Status` | `pending` / `downloaded` / `deferred` / `skipped` / `missing` |

## Asset Download Rules

第一版默认只生成 manifest,不强制下载。

允许下载的条件:

- 用户明确要求下载资源。
- 资源小于仓库约定的体积阈值。
- 资源确认为实现必需,且来源可追溯。
- 下载路径位于 `docs/design/<feature>/assets/` 或后续用户确认的业务项目资源目录。

默认延后的情况:

- Figma 资源只是 screenshot 参考。
- 资源来源不明确。
- 资源可能来自设计系统或业务接口。
- 资源体积较大。
- 资源格式或裁剪规则未确认。

P15 的 `assets/` 目录只是设计实现证据目录,不是业务项目最终静态资源目录。

## Validation Checks

### Markdown Contract Check

最低检查:

- `clarified-requirement.md` 包含 `Open Questions`。
- `ui-understanding.md` 包含 `Page Structure` / `Repeated Patterns` / `Open Questions`。
- `api-mapping.md` 包含 `Data Sources` / `Field Mapping` / `State Mapping` / `Open Questions`。
- `component-mapping.md` 包含分类统计和 `Open Questions`。
- `design-token-patch.md` 包含 token 和 asset references。
- `implementation-spec.md` 包含 coding boundary。
- `open-questions.md` 存在或明确说明尚未进入 Phase E。
- `ui-handoff.md` 如存在,必须声明不是 implementation spec,不授权 coding。
- `design-diff.md` 如存在,必须包含 Recommended Rerun Phases。

### Fixture Contract Check

最低检查:

- 每个新增 skill 至少有 `README.md`、`SKILL.md`、`agents/openai.yaml`。
- 每个有 fixture 的 skill,fixture 至少包含 `README.md` 和 `expected/`。
- suite fixture 必须有 flow map 或 validation checklist。
- fixture README 必须说明是否调用 live Figma MCP。

### Boundary Check

最低检查:

- P12-P15 不修改 `src/`、`app/`、`components/`、`pages/` 等业务代码目录。
- `implementation-spec.md` 不直接引用 raw Figma JSON。
- Phase E handoff 后仍只进入 OpenSpec / planning / task breakdown 等准备阶段。
- 业务代码只能在用户明确确认执行 coding 后开始。
- UI handoff 不等于 Phase E implementation handoff。

### Asset Manifest Check

最低检查:

- 每个 blocking asset 必须有 destination 或 open question。
- `Required=yes` 的资源不能是 `Status=unknown`。
- `Recommended Format=unknown` 的资源必须进入 Open Questions。
- `downloaded` 资源必须有相对路径。
- 大体积二进制资源不能无说明提交。

### Optional LLM-as-Judge

可选用途:

- 比较生成产物和 expected fixture 的语义等价。
- 提示遗漏章节或明显自相矛盾。
- 标记高风险的 open questions。

限制:

- LLM-as-judge 结果只能是 `warn` 或 `risk note`,不能单独判定 pass。
- 结构化 contract check 失败时,LLM-as-judge 不能覆盖失败。
- 不把用户私有业务数据发送给外部服务,除非用户明确确认。

## Orchestrator Integration

`figma-workflow` 可在 v4 工程化区域新增:

```text
Assets / validation:
  [ ] assets-manifest.md (not generated)
  [ ] validation-report.md (not generated)

Actions:
  [A] Generate assets manifest
  [Q] Run validation checks
```

边界:

- assets / validation action 不标记任何 Phase 完成。
- validation warning 不自动阻塞;是否继续由 review gate 和用户决定。
- validation report 不替代人工 review。
- 不因为资源缺失自动写业务代码或下载到业务项目。

---

## Compatibility

- 没有 `assets-manifest.md` 时,现有流程不受影响。
- 没有 `validation-report.md` 时,现有流程不受影响。
- 没有 `ui-handoff.md` 或 `design-diff.md` 时,P15 仍可基于已有产物生成部分结果。
- P15 不改变 Phase E implementation handoff 的语义。

---

## Validation

P15 实现完成后至少验证:

- 能从 `design-token-patch.md` 提取 asset references。
- 能从 `ui-handoff.md` 提取 asset marking gaps。
- 能在没有真实资源下载的情况下生成 `assets-manifest.md`。
- 能生成 `validation-report.md`。
- Markdown contract check 能检测缺失章节。
- Boundary check 能检测业务代码目录变更。
- `ui-handoff.md` 缺失时只给 warning,不阻塞。
- LLM-as-judge 默认跳过,启用时只输出风险提示。
- 不写业务代码。

## Open Questions

- 是否需要第一版实现真实 Figma asset export。建议不默认实现,只在用户明确要求时作为可选动作。
- `assets-manifest.json` 是否和 Markdown 同步生成。建议第一版只生成 Markdown,脚本内部可保留 JSON 中间结构。
- validation checks 是否进入 CI。建议第一版只作为本地命令,等规则稳定后再接 CI。
